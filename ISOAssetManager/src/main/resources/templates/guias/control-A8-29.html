14.2.9
CONTROL Los procesos de pruebas de seguridad deberían definirse e implementarse en el ciclo de vida del desarrollo.
OBJETIVOS Validar si se cumplen los requisitos de SI cuando las aplicaciones o el código se despliegan en el entorno de producción.
GUÍAS

Los nuevos sistemas de información, las actualizaciones y las nuevas versiones deberían probarse y verificarse a fondo durante los procesos de desarrollo. Las pruebas de seguridad deberían ser una parte integral de las pruebas de los sistemas o componentes.

Las pruebas de seguridad deberían realizarse en función de un conjunto de requisitos, que pueden expresarse como funcionales o no funcionales. Las pruebas de seguridad deberían incluir pruebas de:

  1. Funciones de seguridad (por ejemplo, autenticación de usuarios (ver 8.5), restricción de acceso (ver 8.3) y uso de cifrado (ver 8.24));
  2. Codificación segura (ver 8.28);
  3. Configuraciones seguras (ver 8.9, 8.20 y 8.22), incluidas las de los sistemas operativos, los cortafuegos y otros componentes de seguridad.

Los planes de pruebas deberían determinarse mediante un conjunto de criterios. El alcance de las pruebas debería ser proporcional a la importancia, la naturaleza del sistema y el impacto potencial del cambio introducido. El plan de pruebas debería incluir:

  1. Un calendario detallado de actividades y pruebas;
  2. Las entradas y salidas previstas en una serie de condiciones;
  3. Criterios para evaluar los resultados;
  4. Decisión sobre las acciones posteriores que sean necesarias.

La organización puede aprovechar las herramientas automatizadas, por ejemplo, las herramientas de análisis de código o los escáneres de vulnerabilidad y debería verificar la corrección de los defectos relacionados con la seguridad.

En el caso de los desarrollos internos, estas pruebas deberían ser realizadas inicialmente por el equipo de desarrollo. A continuación, deberían realizarse pruebas de aceptación independientes para garantizar que el sistema funciona como se espera y solo como se espera (ver 5.8).

Debería tenerse en cuenta la realización de:

  1. Actividades de revisión del código como elemento relevante para la comprobación de las fallas de seguridad, incluidas las entradas y condiciones no previstas;
  2. Escaneos de vulnerabilidad para identificar configuraciones inseguras y vulnerabilidades del sistema;
  3. Pruebas de penetración para identificar el código y el diseño inseguros.

En el caso de los componentes de desarrollo y compra subcontratados, debería seguirse un proceso de adquisición. Los contratos con el proveedor deberían abordar los requisitos de seguridad identificados (ver 5.20). Los productos y servicios deberían ser evaluados según estos criterios antes de su adquisición.
EJEMPLOS DE IMPLEMENTACIÓN

Las pruebas de seguridad son el "control de calidad" final para asegurarse de que no se introduce vulnerabilidades en producción:

Ejemplo 1: Pruebas de Penetración (Pentesting) antes del "Go-Live"

Antes de lanzar la nueva banca móvil, el banco contrata a una empresa especializada para realizar un "Pentest de Caja Negra". Los pentesters intentan hackear la aplicación simulando ser atacantes externos. Encuentran una vulnerabilidad crítica donde es posible ver el saldo de otros usuarios manipulando la URL. El lanzamiento se pospone hasta que esta vulnerabilidad crítica sea parcheada y verificada mediante un re-test.

Ejemplo 2: Criterios de Aceptación de Seguridad en Historias de Usuario

En el backlog del proyecto, cada Historia de Usuario tiene criterios de aceptación de seguridad ("Definition of Done"). Para la funcionalidad de "Carga de Ficheros", el criterio de aceptación especifica: "El sistema debe rechazar archivos ejecutables (.exe, .sh) y solo permitir .pdf y .jpg". El equipo de QA (Quality Assurance) verifica manualmente este criterio intentando subir archivos maliciosos; si el sistema no los bloquea, la historia se marca como fallida.
OTRA INFO.

Las pruebas deberían realizarse en un entorno de pruebas que se ajuste lo más posible al entorno de producción objetivo para garantizar que el sistema no introduce vulnerabilidades en el entorno de la organización y que las pruebas son fiables (ver 8.31).

Se pueden establecer múltiples entornos de prueba, que se pueden utilizar para diferentes tipos de pruebas (por ejemplo, pruebas funcionales y de rendimiento). Estos diferentes entornos pueden ser virtuales, con configuraciones individuales para simular una variedad de entornos operativos.

También hay que tener en cuenta las pruebas y monitoreo de los entornos de prueba, las herramientas y las tecnologías para garantizar la eficacia de las pruebas. Las mismas consideraciones se aplican a la supervisión de los sistemas de control desplegados en los entornos de desarrollo, prueba y producción. Es necesario tener criterio, guiado por la sensibilidad de los sistemas y los datos, para determinar cuántas capas de metapruebas son útiles.
