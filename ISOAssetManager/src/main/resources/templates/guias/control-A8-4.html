 9.4.5
CONTROL El acceso de lectura y escritura al código fuente, las herramientas de desarrollo y las bibliotecas de software se deberían administrar adecuadamente.
OBJETIVOS Para evitar la introducción de funciones no autorizadas, evite cambios no intencionados o malintencionados y mantenga la confidencialidad de la propiedad intelectual valiosa.
GUÍAS

El acceso al código fuente y a los elementos asociados (como diseños, especificaciones, planes de verificación y planes de validación) y a las herramientas de desarrollo (por ejemplo, compiladores, constructores, herramientas de integración, plataformas de prueba y entornos) deberían controlarse rigurosamente.

Para el código fuente, esto se puede lograr al controlar el almacenamiento centralizado de dicho código, preferiblemente en el sistema de gestión del código fuente.

El acceso de lectura y escritura al código fuente puede variar en función de la función del personal. Por ejemplo, el acceso de lectura al código fuente se puede proporcionar ampliamente dentro de la organización, pero el acceso de escritura al código fuente solo está disponible para el personal con privilegios o los propietarios designados. Cuando varios desarrolladores de una organización utilizan componentes de código, se debería implementar el acceso de lectura a un repositorio de código centralizado. Además, si se utiliza código de código abierto o componentes de código de terceros dentro de una organización, se puede proporcionar acceso de lectura a dichos repositorios de código externo. Sin embargo, el acceso de escritura debería seguir estando restringido.

Se deberían considerar las siguientes directrices para controlar el acceso a las bibliotecas de origen de programas a fin de reducir el potencial de corrupción de los programas informáticos:

  1. Administrar el acceso al código fuente del programa y a las bibliotecas de origen del programa según los procedimientos establecidos;
  2. Conceder acceso de lectura y escritura al código fuente en función de las necesidades empresariales y gestionar para abordar los riesgos de alteración o uso indebido y de acuerdo con los procedimientos establecidos;
  3. Actualizar el código fuente y los elementos asociados y conceder acceso al código fuente de acuerdo con los procedimientos de control de cambios (ver 8.32) y realizarlo solo después de recibir la autorización adecuada;
  4. No conceder a los desarrolladores acceso directo al repositorio de código fuente, sino a través de herramientas de desarrollo que controlan las actividades y autorizaciones en el código fuente;
  5. Mantener listas de programas en un entorno seguro, donde el acceso de lectura y escritura debería gestionarse y asignarse de forma adecuada;
  6. Mantener un registro de auditoría de todos los accesos y de todos los cambios realizados en el código fuente.

Si se pretende publicar el código fuente del programa, deberían considerarse controles adicionales para garantizar su integridad (por ejemplo, la firma digital).
EJEMPLOS DE IMPLEMENTACIÓN

El código fuente es la "joya de la corona" de una empresa de software y un mapa para los atacantes:

Ejemplo 1: Protección de Ramas (Branch Protection)

En el repositorio Git de la empresa, la rama `master` o `main` está protegida. Ningún desarrollador, ni siquiera los senior, pueden hacer "push" directo. Todos los cambios deben venir a través de una "Pull Request" que requiere obligatoriamente la aprobación de al menos otro desarrollador (Code Review) y pasar todos los tests automáticos (CI/CD) antes de fusionarse.

Ejemplo 2: Minimización de Acceso al Repositorio

Los desarrolladores del equipo "Frontend" no tienen acceso de lectura al repositorio "Backend-Pagos-Core". Esto sigue el principio de mínimo privilegio: si la cuenta de un desarrollador frontend es comprometida, el atacante no podrá robar ni analizar la lógica crítica de procesamiento de pagos.
OTRA INFO. Si el acceso al código fuente no se controla correctamente, el código fuente se puede modificar o algunos datos del entorno de desarrollo (por ejemplo, copias de datos de producción, detalles de configuración) pueden ser recuperados por personas no autorizadas.
