 14.2.1
CONTROL Deberían establecerse y aplicarse normas para el desarrollo seguro de software y sistemas.
OBJETIVOS Garantizar que la SI se diseña e implementa dentro del ciclo de vida de desarrollo seguro del software y los sistemas.
GUÍAS

El desarrollo seguro es un requisito para crear un servicio, una arquitectura, un software y un sistema seguros. Para lograr esto, se deberían tomar en consideración los siguientes factores:

  1. Separación de entornos de desarrollo, pruebas y entornos de producción (ver el numeral 8.31);
  2. Guía sobre la seguridad en el ciclo de vida de desarrollo de software:
    1. Seguridad en la metodología de desarrollo de software (ver 8.28 y 8.27);
    2. Directrices de codificación seguras para cada lenguaje de programación utilizado (ver 8.28);
  3. Requisitos de seguridad en la fase de especificación y diseño (ver 5.8);
  4. Puntos de control de seguridad en los proyectos (ver 5.8);
  5. Pruebas de sistema y seguridad, como pruebas de regresión, análisis de código y pruebas de penetración (ver 8.29);
  6. Repositorios seguros para el código fuente y la configuración (ver 8.4 y 8.9);
  7. Seguridad en el control de versiones (ver 8.32);
  8. Conocimientos y formación necesarios sobre seguridad de las aplicaciones (ver el numeral 8.28);
  9. Capacidad de los desarrolladores para prevenir, encontrar y corregir vulnerabilidades (ver 8.28);
  10. Requisitos de licencia y alternativas para garantizar soluciones rentables y evitar futuros problemas de licencia (ver 5.32).

Si se subcontrata el desarrollo, la organización debería obtener la garantía de que el proveedor cumple con las reglas de desarrollo seguro de la organización (ver 8.30).
EJEMPLOS DE IMPLEMENTACIÓN

La seguridad no es una "capa de pintura" que se añade al final; debe estar en los cimientos del edificio (Security by Design):

Ejemplo 1: Integración DevSecOps (SAST/DAST)

En el pipeline de despliegue continuo (CI/CD) de la empresa, se integran escáneres automáticos. Cada vez que un desarrollador hace un "commit", una herramienta SAST (como SonarQube) analiza el código en busca de vulnerabilidades estáticas, y una herramienta DAST escanea la aplicación en ejecución. Si se detecta una vulnerabilidad crítica (como una inyección SQL), la compilación falla automáticamente y se impide el despliegue a producción.

Ejemplo 2: Modelado de Amenazas (Threat Modeling)

Antes de escribir una sola línea de código para el nuevo portal de clientes, el equipo de arquitectura realiza una sesión de modelado de amenazas utilizando metodologías como STRIDE. Identifican que la función de "recordar contraseña" podría ser abusada para robar cuentas. Como resultado, deciden implementar autenticación multifactor (MFA) desde el diseño inicial, en lugar de intentar parchearlo después del lanzamiento.
OTRA INFO. El desarrollo también puede tener lugar dentro de aplicaciones, como aplicaciones de ofimática, secuencias de comandos, exploradores y bases de datos.
