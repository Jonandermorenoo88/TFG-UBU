 14.3.1
CONTROL La información para las pruebas debería seleccionarse, protegerse y gestionarse adecuadamente.
OBJETIVOS Garantizar la pertinencia de las pruebas y la protección de la información operativa utilizada para las mismas.
GUÍAS

La información para las pruebas debería seleccionarse para garantizar la fiabilidad de los resultados de las pruebas y la confidencialidad de la información operativa pertinente. La información sensible (incluida la información de identificación personal) no debería copiarse en los entornos de desarrollo y pruebas (ver 8.31).

Las siguientes directrices deberían aplicarse para proteger las copias de la información operativa, cuando se utilicen con fines de prueba, tanto si el entorno de prueba se construye internamente como en un servicio en la nube:

  1. Aplicar a los entornos de prueba los mismos procedimientos de control de acceso que se aplican a los entornos operativos;
  2. Tener una autorización independiente cada vez que se copie información operativa en un entorno de pruebas;
  3. Registrar la copia y el uso de la información operativa para proporcionar un registro de auditoría;
  4. Proteger la información sensible mediante su eliminación o enmascaramiento (ver 8.11) si se utiliza para las pruebas;
  5. Eliminar adecuadamente (ver 8.10) la información operativa de un entorno de prueba inmediatamente después de que se haya completado la prueba para evitar el uso no autorizado de la información de prueba.

La información de las pruebas debería almacenarse de forma segura (para evitar su manipulación, que puede dar lugar a resultados no válidos) y utilizarse únicamente con fines de prueba.
EJEMPLOS DE IMPLEMENTACIÓN

En pruebas, necesitas datos realistas, pero usar datos reales es un riesgo. La clave es el equilibrio entre realismo y privacidad:

Ejemplo 1: Generación de Datos Sintéticos

En lugar de copiar la base de datos de producción con datos reales de clientes, el equipo de QA utiliza una herramienta (como Mockaroo o Faker) para generar 10,000 registros de "clientes ficticios". Estos datos respetan la estructura y la lógica de negocio (NIFs válidos, códigos postales reales) pero no corresponden a ninguna persona real, permitiendo pruebas funcionales completas sin riesgo de fuga de datos personales.

Ejemplo 2: Enmascaramiento y Borrado Seguro

Cuando es imprescindible usar datos de producción para depurar un error complejo, se sigue un procedimiento estricto: se solicita aprobación al CISO, se copia solo el subconjunto de datos necesario a un entorno aislado, se aplica una herramienta de enmascaramiento para ocultar campos sensibles (como números de tarjeta de crédito o diagnósticos médicos) y, una vez finalizada la prueba, los datos se eliminan mediante borrado seguro.
OTRA INFO. Las pruebas de sistemas y de aceptación pueden requerir volúmenes considerables de información de prueba que se acerquen lo más posible a la información operativa.
